Intégrer l'API

Comment authentifier vos appels, un exemple d'intégration complet de bout en bout, et la liste des codes d'erreur que votre backend doit savoir gérer.

Comment authentifier vos appels

Deux mécanismes existent, ne les confondez pas :

MécanismeFormatUtilisé pour
Clé APIbn_live_...Appels depuis votre backend : /notify, /notifications/feed, /preferences...
JWTeyJ0eXAi...Gestion de compte depuis votre dashboard : règles, canal de secours, statistiques

Dans les deux cas, la clé se passe en en-tête HTTP :

Authorization: Bearer bn_live_9fK2mQ7Z...
Sécurité : la clé API ne doit jamais être exposée côté navigateur — uniquement depuis votre serveur backend.

Tester avant d'intégrer (Postman)

Avant d'écrire le moindre code, testez vos appels dans Postman (ou tout client HTTP équivalent). Ça permet de valider votre clé API, vos règles et le format de vos données sans redémarrer votre application à chaque essai.

Prérequis : une règle doit déjà exister pour l'event_code que vous testez (ex : payment_confirmed), sinon /notify renvoie 404 REGLE_NOT_FOUND. Créez-la depuis le dashboard (onglet Règles) ou via POST /api/regles avant de continuer.

  1. Créer une nouvelle requête

    Méthode POST, URL : https://votre-cdn.com/api/notify

  2. Renseigner les en-têtes (onglet Headers)

    • Authorization : Bearer bn_live_... (votre clé API, récupérée depuis le dashboard)
    • Content-Type : application/json

  3. Renseigner le corps (onglet Body → raw → JSON)

    {
  "event": "payment_confirmed",
  "user_id": "USR123",
  "to": {"email": "client@exemple.com"},
  "variables": {"amount": "5000", "name": "Aïssatou"}
}

  4. Envoyer et vérifier la réponse

    Un 201 avec la liste des notification_ids confirme que tout est correctement configuré. Une erreur ? Consultez la table des codes d'erreur ci-dessous pour l'interpréter avant de reproduire l'appel dans votre code.

Répétez cette vérification pour chaque endpoint que vous comptez intégrer (/preferences, /notifications/feed...) — c'est plus rapide de corriger un en-tête ou un champ manquant dans Postman que dans votre application.

Exemple d'intégration complet

Scénario : votre application veut notifier un utilisateur quand son paiement est confirmé.

Étape 1 — Créer la règle une seule fois (setup)

curl -X POST \
  -H "Authorization: Bearer {jwt}" \
  -H "Content-Type: application/json" \
  -d '{
    "event_code": "payment_confirmed",
    "message_template": "Paiement de {amount} FCFA confirmé, merci {name} !",
    "email_actif": true,
    "socket_actif": true
  }' \
  https://votre-cdn.com/api/regles

Étape 2 — Déclencher l'événement à chaque paiement (runtime)

curl -X POST \
  -H "Authorization: Bearer bn_live_9fK2mQ7Z..." \
  -H "Content-Type: application/json" \
  -d '{
    "event": "payment_confirmed",
    "user_id": "USR123",
    "to": {"email": "client@exemple.com"},
    "variables": {"amount": "5000", "name": "Aïssatou"}
  }' \
  https://votre-cdn.com/api/notify

// Réponse 201
{
  "message": "Notification(s) mise(s) en queue.",
  "canaux": ["email", "socket"],
  "notification_ids": [142, 143]
}

Étape 3 — Gérer la réponse dans votre code

const res = await fetch('https://votre-cdn.com/api/notify', {
  method: 'POST',
  headers: {
    Authorization: `Bearer ${BABELNOTIF_API_KEY}`,
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    event: 'payment_confirmed',
    user_id: user.id,
    to: { email: user.email },
    variables: { amount: payment.amount, name: user.firstName },
  }),
});

if (!res.ok) {
  const erreur = await res.json();
  // voir la table des codes d'erreur ci-dessous
  console.error(erreur.code, erreur.message);
}
