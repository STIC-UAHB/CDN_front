Envoyer des notifications

POST /api/notify

L'endpoint central : déclenche l'envoi d'une notification pour un événement donné. C'est le seul appel que votre backend a besoin de connaître pour toute la logique d'envoi.

Ce que fait BabelNotif quand vous appelez /notify

  1. Valide les données reçues (event, user_id, to requis)
  2. Vérifie votre clé API et que votre compte est actif
  3. Cherche la règle active correspondant à event
  4. Vérifie les préférences de user_id pour cette règle (un utilisateur peut avoir désactivé un canal, ou la règle entière)
  5. Détermine la liste finale des canaux actifs
  6. Compile le message en remplaçant les {variable} du template par les valeurs de variables
  7. Crée une notification par canal actif et l'envoie (ou la planifie si scheduled_at est fourni)
  8. Incrémente votre quota utilisé

Paramètres

ChampTypeRequisDescription
eventstringouiCode de l'événement, doit correspondre à une règle active
user_idstringouiIdentifiant de votre utilisateur final
to.emailstringconditionnelAdresse email — requis si to.sms absent
to.smsstringconditionnelNuméro de téléphone — requis si to.email absent
variablesobjectnonValeurs injectées dans le message_template de la règle
scheduled_atstring (ISO date)nonDate future — la notification est planifiée au lieu d'être envoyée immédiatement
Important : même si votre règle n'active que push ou socket, vous devez quand même fournir to.email ou to.sms — c'est ce qui sert de destination si BabelNotif doit escalader vers un canal de secours (voir Canal de secours).

Exemple

# Envoi immédiat
curl -X POST \
  -H "Authorization: Bearer bn_live_..." \
  -H "Content-Type: application/json" \
  -d '{
    "event": "payment_confirmed",
    "user_id": "USR123",
    "to": {"email": "client@exemple.com"},
    "variables": {"amount": "5000", "name": "Aïssatou"}
  }' \
  https://votre-cdn.com/api/notify
# Envoi planifié
curl -X POST \
  -H "Authorization: Bearer bn_live_..." \
  -H "Content-Type: application/json" \
  -d '{
    "event": "subscription_reminder",
    "user_id": "USR123",
    "to": {"email": "client@exemple.com"},
    "scheduled_at": "2026-07-10T09:00:00Z"
  }' \
  https://votre-cdn.com/api/notify
// Réponse 201
{
  "message": "Notification(s) mise(s) en queue.",
  "canaux": ["email", "socket"],
  "notification_ids": [142, 143],
  "scheduled_at": null
}

Cas particuliers

  • Si aucun canal n'est actif pour cet utilisateur et cette règle (préférences désactivées), la réponse est 200 avec un message informatif — aucune erreur, aucune notification créée.
  • Le canal push est automatiquement ignoré si l'utilisateur n'a pas de device_token enregistré (voir Gérer les préférences).
  • Les codes d'erreur possibles sont listés dans Intégrer l'API.
