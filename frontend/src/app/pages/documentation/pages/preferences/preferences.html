Gérer les préférences utilisateur

Les préférences permettent à chacun de vos utilisateurs finaux de personnaliser son expérience : désactiver un canal précis, désactiver une règle entière, ou enregistrer son device_token pour recevoir des notifications push.

C'est vous qui construisez l'interface. BabelNotif ne fournit aucun écran de préférences pour vos utilisateurs finaux — c'est à votre application d'exposer un réglage (ex : une page "Notifications" dans vos paramètres de compte) où l'utilisateur choisit ses canaux, puis d'appeler l'endpoint ci-dessous à chaque modification pour la transmettre à BabelNotif.

Comment les préférences priment sur les règles

À l'envoi, pour chaque canal, BabelNotif applique cet ordre de priorité :

  1. Si la règle est non_desactivable → les préférences sont ignorées, le default de la règle s'applique toujours.
  2. Sinon, si l'utilisateur a une préférence enregistrée pour ce canal → elle prévaut.
  3. Sinon → le default de la règle s'applique.

Mettre à jour une préférence

POST /api/preferences
ChampTypeDescription
user_idstringL'utilisateur concerné
event_codestringLa règle concernée (doit exister)
rule_actifbooleanInterrupteur maître — false désactive toute la règle pour cet utilisateur
sms_actif / email_actif / push_actif / socket_actifbooleanOverride canal par canal (n'envoyer que si différent du default de la règle)
device_tokenstringToken Firebase FCM de l'appareil, nécessaire pour recevoir le canal push

Exemple — désactiver le SMS pour un événement précis

curl -X POST \
  -H "Authorization: Bearer bn_live_..." \
  -H "Content-Type: application/json" \
  -d '{
    "user_id": "USR123",
    "event_code": "payment_confirmed",
    "sms_actif": false
  }' \
  https://votre-cdn.com/api/preferences

Exemple — désactiver complètement une règle non critique

curl -X POST \
  -H "Authorization: Bearer bn_live_..." \
  -H "Content-Type: application/json" \
  -d '{"user_id": "USR123", "event_code": "weekly_report", "rule_actif": false}' \
  https://votre-cdn.com/api/preferences
Note : une règle marquée non_desactivable (typiquement un OTP de sécurité) ignore rule_actif — l'utilisateur ne peut pas la couper.
D'où vient le device_token ? Ce n'est pas BabelNotif qui le génère — voir Intégrer le Push pour le guide complet (SDK Firebase, permission navigateur, service worker).

Lire les préférences d'un utilisateur

GET /api/preferences/{user_id}

Renvoie toutes les préférences enregistrées pour cet utilisateur, avec la règle associée.

  -H "Authorization: Bearer bn_live_..." \
  https://votre-cdn.com/api/preferences/USR123
