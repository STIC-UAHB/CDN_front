Intégrer le WebSocket

Ce guide s'adresse au développeur qui construit le frontend de votre application (pas le dashboard BabelNotif) — celui qui doit afficher les notifications en temps réel à vos utilisateurs.

Pourquoi le WebSocket est différent des autres canaux

SMS et email sont livrés directement par un opérateur (Orange, SMTP) : une fois envoyés, votre utilisateur les verra, connecté ou non. Le canal socket est différent : la notification n'est reçue que si votre frontend est connecté au moment de l'envoi. D'où l'importance des 3 étapes ci-dessous, à implémenter toutes les trois.

Principe important : votre clé API ne doit jamais atteindre le navigateur de votre utilisateur final. Toutes les requêtes vers BabelNotif qui nécessitent la clé API (autorisation du channel, confirmation de lecture, rattrapage des notifs) doivent passer par votre propre backend, qui la détient en secret et la relaie. Votre frontend n'appelle jamais BabelNotif directement pour ça — il appelle vos propres routes, qui elles-mêmes appellent BabelNotif.

Étape 1 — Connecter le navigateur au channel

Côté frontend, votre navigateur se connecte au serveur Reverb de BabelNotif :

npm install laravel-echo pusher-js
// Côté frontend (navigateur)
import Echo from 'laravel-echo';
import Pusher from 'pusher-js';

const echo = new Echo({
  broadcaster: 'reverb',
  key: '6j7g15fjzc4glckgiqyo',   // clé publique du serveur Reverb, la même pour tous les clients BabelNotif — pas secrète
  wsHost: 'dazzling-commitment-production-fa21.up.railway.app',
  wssPort: 443,
  forceTLS: true,
  authEndpoint: '/ws-auth',      // route de VOTRE PROPRE backend, pas celle de BabelNotif
});

Le reverb_app_key est une valeur publique commune à tous les clients BabelNotif (comme la "clé app" de Pusher) — elle identifie le serveur Reverb, pas votre compte. Elle ne remplace pas votre clé API, qui elle reste secrète.

Votre backend fait le pont (proxy d'authentification)

Le navigateur appelle /ws-auth sur votre serveur. Votre backend relaie cette requête vers /broadcasting/auth de BabelNotif, en y ajoutant votre clé API — que le navigateur ne voit jamais : 

// Côté backend (le vôtre) — exemple générique, à adapter à votre stack
app.post('/ws-auth', async (req, res) => {
  const reponse = await fetch('/api/broadcasting/auth', {
    method: 'POST',
    headers: {
      Authorization: `Bearer ${BABELNOTIF_API_KEY}`,  // gardée côté serveur uniquement
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      channel_name: req.body.channel_name,
      socket_id: req.body.socket_id,
    }),
  });
  const data = await reponse.json();
  res.json(data);
});

Étape 2 — Écouter et confirmer la lecture

Chaque utilisateur a son propre channel privé, au format notifications.{client_id}.user_{user_id}. Une fois l'authentification (étape 1) en place, écoutez les notifications : 

// Côté frontend (navigateur)
echo.private(`notifications.${clientId}.user_${userId}`)
  .listen('NotificationEnvoyee', (data) => {
    afficherNotificationDansLUI(data.message);

    // dès que l'utilisateur la voit vraiment, on informe VOTRE backend :
    fetch(`/notifications/${data.id}/lu`, { method: 'POST' });
  });
POST /api/notifications/{id}/lu

Votre route /notifications/{id}/lu (côté backend) relaie simplement l'appel vers BabelNotif avec votre clé API, comme à l'étape 1 :

// Côté backend (le vôtre)
app.post('/notifications/:id/lu', async (req, res) => {
  await fetch(`/api/notifications/${req.params.id}/lu`, {
    method: 'POST',
    headers: { Authorization: `Bearer ${BABELNOTIF_API_KEY}` },
  });
  res.sendStatus(200);
});
Pourquoi confirmer la lecture ? Sans cet appel, BabelNotif considère la notification comme non lue et, après un délai, bascule automatiquement vers un canal de secours (email par défaut) — voir Canal de secours.

Étape 3 — Rattraper les notifications manquées

Le WebSocket seul ne suffit pas : si votre utilisateur n'était pas connecté au moment de l'envoi, il rate le message. Au chargement de votre application (et idéalement à chaque reconnexion), votre frontend appelle votre propre backend, qui relaie vers le "feed" de BabelNotif :

GET /api/notifications/feed
// Côté frontend (navigateur)
const res = await fetch(`/notifications/manquees?user_id=${userId}`);
const { notifications } = await res.json();
notifications.forEach(afficherNotificationDansLUI);
// Côté backend (le vôtre)
app.get('/notifications/manquees', async (req, res) => {
  const reponse = await fetch(
    `/api/notifications/feed?user_id=${req.query.user_id}`,
    { headers: { Authorization: `Bearer ${BABELNOTIF_API_KEY}` } }
  );
  res.json(await reponse.json());
});

Cet endpoint ne renvoie que les notifications socket dont lu_at est encore vide — dès qu'une notification est confirmée via /lu, elle disparaît du feed.

Intégration mobile (Flutter, Android, iOS natif)

Le principe est exactement le même que pour le web : votre backend proxy l'authentification, votre app se connecte à Reverb. Seule la librairie change — laravel-echo et pusher-js sont spécifiques au navigateur. Reverb parlant nativement le protocole Pusher, utilisez l'équivalent natif de votre plateforme :

PlateformeLibrairie
Flutterpusher_channels_flutter
Android natif (Kotlin/Java)pusher-websocket-java
iOS natif (Swift)PusherSwift

Exemple — Flutter

flutter pub add pusher_channels_flutter
// Côté app Flutter
import 'package:pusher_channels_flutter/pusher_channels_flutter.dart';

final pusher = PusherChannelsFlutter.getInstance();

Future<void> connecterWebSocket(String clientId, String userId) async {
  await pusher.init(
    apiKey: '6j7g15fjzc4glckgiqyo', // clé publique du serveur Reverb BabelNotif, la même que pour le web
    cluster: '',
    onAuthorizer: (channelName, socketId, options) async {
      // relayé par VOTRE backend, comme pour le web — jamais votre clé API dans l'app
      final res = await http.post(
        Uri.parse('https://mon-backend.com/ws-auth'),
        body: {'channel_name': channelName, 'socket_id': socketId},
      );
      return jsonDecode(res.body);
    },
  );

  await pusher.connect();
  await pusher.subscribe(
    channelName: 'private-notifications.$clientId.user_$userId',
    onEvent: (event) {
      if (event.eventName == 'NotificationEnvoyee') {
        final data = jsonDecode(event.data);
        afficherNotification(data['message']);

        // confirmer la lecture, comme pour le web
        http.post(Uri.parse('https://mon-backend.com/notifications/${data['id']}/lu'));
      }
    },
  );
}
Réglage useTLS/scheme : adaptez selon votre environnement — false en local (Reverb servi en http), true une fois votre serveur Reverb exposé en https en production.

Le nom du channel (private-notifications.{clientId}.user_{userId}), l'événement (NotificationEnvoyee) et les routes de rattrapage (/feed) sont strictement identiques à la version web — seule la syntaxe de connexion change d'une plateforme à l'autre.

Récapitulatif du flux complet

  1. Votre backend appelle POST /api/notify
  2. BabelNotif diffuse l'event NotificationEnvoyee sur le channel privé de l'utilisateur
  3. Le navigateur s'authentifie sur ce channel via votre route proxy /ws-auth (votre backend, votre clé API)
  4. Si le navigateur est connecté : il reçoit la notif en direct et confirme la lecture via votre backend, qui relaie vers /lu
  5. Au chargement (ou reconnexion), le navigateur passe par votre backend pour appeler /feed et rattraper ce qui aurait pu être manqué
  6. Si jamais confirmée comme lue après le délai configuré, BabelNotif bascule automatiquement vers le canal de secours

Comprenez le mécanisme d'escalade dans Canal de secours.
