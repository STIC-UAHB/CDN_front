Intégrer le Push

Ce guide s'adresse au développeur qui construit le frontend de votre application (pas le dashboard BabelNotif) — celui qui doit obtenir un device_token auprès de Firebase et l'enregistrer auprès de BabelNotif pour que le canal push fonctionne.

Web uniquement pour l'instant. L'intégration ci-dessous couvre les applications web (navigateur). Le push pour applications mobiles natives (Flutter, Android, iOS) n'est pas encore documenté — contactez-nous si c'est votre besoin.

D'où vient le device_token ?

Le device_token n'est pas généré par BabelNotif — c'est votre application qui le récupère, via le SDK Firebase, en s'enregistrant auprès du projet Firebase de BabelNotif (pas le vôtre). C'est le même principe que la clé Reverb partagée pour le WebSocket : tous les clients BabelNotif utilisent la même configuration publique ci-dessous, parce que c'est le backend BabelNotif qui envoie effectivement les push.

Public, pas secret : contrairement à la clé API, cette configuration Firebase est faite pour être intégrée telle quelle dans votre code frontend (navigateur).

Étape 2 — Demander la permission et récupérer le token

// Côté frontend (navigateur)
async function activerNotificationsPush(userId) {
  const permission = await Notification.requestPermission();
  if (permission !== 'granted') return;

  // nécessite un service worker enregistré à la racine (voir plus bas)
  const registration = await navigator.serviceWorker.ready;

  const token = await getToken(messaging, {
    vapidKey: 'BFjxK3Bx79oz4rFwjNS96AMoVAUg1YCu5uovtDNqxP5ae7JHKjHqfjxnZBucIgdvvD8F86uVYYd7-yQ3GqfQYBM',
    serviceWorkerRegistration: registration,
  });

  // on envoie le token à VOTRE PROPRE backend, jamais directement à BabelNotif
  await fetch('/mon-backend/enregistrer-push', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ user_id: userId, device_token: token }),
  });
}

Étape 3 — Recevoir en arrière-plan (service worker)

Le Web Push a besoin d'un service worker à la racine de votre domaine pour afficher une notification quand votre onglet n'est pas au premier plan (ou fermé, tant que le navigateur tourne). Créez un fichier firebase-messaging-sw.js à la racine de votre site : 

// firebase-messaging-sw.js — à la racine de votre domaine
importScripts('https://www.gstatic.com/firebasejs/10.13.2/firebase-app-compat.js');
importScripts('https://www.gstatic.com/firebasejs/10.13.2/firebase-messaging-compat.js');

firebase.initializeApp({
  apiKey: "AIzaSyD_L3NCOV2MUJrXJVFJSXUj54pglpcKQh0",
  authDomain: "babelnotif.firebaseapp.com",
  projectId: "babelnotif",
  storageBucket: "babelnotif.firebasestorage.app",
  messagingSenderId: "207367888684",
  appId: "1:207367888684:web:82338d8160e718f6729689",
});

const messaging = firebase.messaging();

messaging.onBackgroundMessage((payload) => {
  const { title, body } = payload.notification || {};
  self.registration.showNotification(title || 'Notification', { body: body || '' });
});
Premier plan vs arrière-plan : quand votre onglet est actif, c'est plutôt onMessage() (dans votre code frontend normal) qui reçoit la notification — à vous de l'afficher dans votre UI. Le service worker ci-dessus ne prend le relais que quand l'onglet n'est pas au premier plan.

Étape 4 — Votre backend relaie le token à BabelNotif

Comme pour l'authentification WebSocket, c'est votre backend (qui détient la clé API) qui fait l'appel réel :

// Côté backend (le vôtre)
app.post('/mon-backend/enregistrer-push', async (req, res) => {
  await fetch('/api/preferences', {
    method: 'POST',
    headers: {
      Authorization: `Bearer ${BABELNOTIF_API_KEY}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      user_id: req.body.user_id,
      event_code: 'payment_confirmed', // la règle concernée
      device_token: req.body.device_token,
    }),
  });
  res.sendStatus(200);
});

Ce même endpoint est documenté en détail dans Gérer les préférences — c'est POST /api/preferences avec le champ device_token.

Sans device_token enregistré, le canal push est automatiquement ignoré à l'envoi — aucune erreur, la notification est juste sautée pour ce canal.
