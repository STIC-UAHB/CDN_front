Gérer les règles

Une règle représente un événement de votre application qui doit être signalé à un utilisateur (ex : un paiement reçu, un compte qui expire, un abonnement qui se termine). Pour chaque règle, vous définissez sur quels canaux l'informer par défaut et avec quel message. L'ensemble de vos règles forme le catalogue des événements que BabelNotif sait notifier — c'est la première chose à configurer avant d'appeler /notify.

Vous pouvez créer et consulter vos règles directement depuis le dashboard (onglet Règles). L'endpoint ci-dessous fait exactement la même chose, requiert le JWT (le même jeton obtenu à la connexion) — c'est donc une action liée à votre session dashboard, pas une automatisation depuis votre code. Pour synchroniser vos règles depuis un script ou un pipeline CI/CD sans dépendre d'un JWT, utilisez plutôt l'import en masse ci-dessous, qui accepte aussi la clé API.

Créer une règle par API

POST /api/regles JWT
ChampTypeDescription
event_codestringIdentifiant unique de l'événement (utilisé ensuite dans /notify)
message_templatestringTexte du message, avec des variables entre accolades : {variable}
descriptionstringNote libre pour vous y retrouver (optionnel)
sms_actifbooleanCanal SMS actif par défaut
email_actifbooleanCanal email actif par défaut
push_actifbooleanCanal push actif par défaut
socket_actifbooleanCanal WebSocket actif par défaut
non_desactivablebooleanSi vrai, l'utilisateur final ne peut jamais désactiver ce canal via ses préférences (ex : un code de sécurité)
curl -X POST \
  -H "Authorization: Bearer {jwt}" \
  -H "Content-Type: application/json" \
  -d '{
    "event_code": "payment_confirmed",
    "message_template": "Paiement de {amount} FCFA confirmé, merci {name} !",
    "sms_actif": false,
    "email_actif": true,
    "push_actif": false,
    "socket_actif": true,
    "non_desactivable": false
  }' \
  /api/regles

// Réponse 201
{
  "id": 20,
  "event_code": "payment_confirmed",
  "message_template": "Paiement de {amount} FCFA confirmé, merci {name} !",
  "email_actif": true,
  "socket_actif": true,
  "actif": true
}

Télécharger un modèle

GET /api/regles/template JWT

Télécharge un fichier JSON vide au bon format, pratique pour préparer un import en masse.

Importer plusieurs règles en masse

POST /api/regles/import JWT ou clé API

Accepte un tableau de règles — utile pour synchroniser vos règles depuis un fichier de config versionné (CI/CD). Chaque règle est créée ou mise à jour si event_code existe déjà.

curl -X POST \
  -H "Authorization: Bearer {jwt_ou_api_key}" \
  -H "Content-Type: application/json" \
  -d '[
    {"event_code": "payment_confirmed", "message_template": "...", "email_actif": true},
    {"event_code": "payment_failed", "message_template": "...", "sms_actif": true}
  ]' \
  /api/regles/import
Nécessite un abonnement actif (l'essai gratuit compte) — sinon 402 no_active_subscription.

Et ensuite ?

Une fois vos règles créées, personnalisez le comportement par utilisateur avec les préférences, ou déclenchez l'envoi avec /notify.
